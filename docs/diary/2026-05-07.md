# 2026-05-07

## Sezení 1 — Definice projektu, scaffold

### Diskuse

- Definice projektu: webová sandbox simulace dynamiky pixelů.
- Cíl: experiment a zábava, žádný fixní deadline ani feature scope.
- Konvence sezení: otázky a odpovědi se číslují `Q1, Q2…` / `A1, A2…`.
- Mód spolupráce: maximální skepticismus a kriticismus ze strany AI.
- Zatím bez `GLOSSARY.md` — pojmy se postupně budují, glosář vznikne, až bude třeba.

### Rozhodnutí

**Stack:**
- Vite + TypeScript (strict, `noUncheckedIndexedAccess`)
- Svelte 5 (UI panely, runes)
- Rapier 2D (`@dimforge/rapier2d-compat` 0.19.3) — rigid body fyzika
- WebGL2 + instanced quads (raw, bez render libs)
- gl-matrix (jediná math závislost)

**Klíčový pivot:** uživatel správně poukázal, že potřebujeme FW na 2D kolize. Původní návrh (custom XPBD + shape matching) byl over-engineered. Místo toho:
- Každý pixel = 1 rigid body s nativní rotací.
- Slepování = `FixedJoint` (později breakable).
- Pružnost = `DistanceJoint` se spring koeficientem.
- Rozbití = remove joint při překročení impulse threshold.

**Konvence:**
- Identifikátory anglicky, komentáře česky.
- Bez `T` prefixu (Delphi konvence se neaplikuje).
- `kebab-case.ts` filenames.

**Pixel struktura ve FVP:**
```ts
type Pixel = {
  id: number;
  body: RigidBody;   // Rapier drží x, y, vx, vy, r, rs
  m: number;         // konstantní 1, ale strukturně přítomné
};
```
Záměr: `m` připravený teď, abychom při fázi 4 nepřepisovali API.

### Kód

Scaffold projektu (Vite + Svelte 5 + TS), instalace deps, vytvořena adresářová struktura `src/{sim,render,input,ui}`. FVP implementace:
- `src/sim/physics.ts` — Rapier World wrapper, `spawnPixel(x, y)` s náhodnými počátečními rychlostmi a rotací.
- `src/render/gl.ts` — WebGL2 instanced renderer.
- `src/render/camera.ts` — 2D ortografická projekce, `screenToWorld` helper.
- `src/input/keyboard.ts` — set stisknutých kláves.
- `src/ui/App.svelte` — layout 3 sloupců, FPS + pixel counter, roadmap v pravém panelu.
- GitHub Actions deploy na Pages, MIT licence.

### Kudos / Censure

**Kudos! uživatel:** Včasné upozornění na potřebu kolizního FW — ušetřilo to týdny psaní vlastního shape matchingu. Skvělé řízení AI mimo over-engineering.

**Censure! sobě:** První návrh (XPBD + shape matching pro slepené clustery) byl ambiciózní v matematice tam, kde existující engine řeší problém přímočaře. Default to libraries first, custom solvers last.

## Sezení 2 — Fáze 2: gravitace + UI + pivot z Rapier integrátoru

### Diskuse

- Pixelová gravitace: párová Newton, centrální, nebo k těžišti? Uživatel zvolil **párovou**.
- Object dataset architektura — pre-computed centroid + cached state pro composite objekty (fáze 3+).
- Zkratka pro úhlovou rychlost objektu: `rs` (izomorfismus s `Pixel.rs`), ne VR/omega/W.
- Bug report: "stále se rozbíhají, gravitační prak". Diagnóza přes ∑P (klesá), ∑L (diverguje na ~10⁵) → integrátor je viník.
- Levý panel: STATS + FACTS (championi). Pravý panel: SETTINGS (G, H) + COMMANDS.
- `ε` (Plummer softening) je numerický regularizátor, ne fyzikální parametr → schováno z UI.
- Spawn linvel/angvel intervaly přesunuty do `params.ts` jako konstanty programu.

### Rozhodnutí

- **Solver pivot:** manuální symplektický Euler pro FVP, **Rapier integrátor obejit**. Rapier RigidBody zůstává jen jako kontejner stavu. Až ve fázi 3 (joints, kolize) se Rapier step() vrací — kontakty pak fyzicky brání slingshotům.
- **Důvod:** Rapier addForce + step v čistě gravitační simulaci porušoval Newton 3 a zachování úhlové hybnosti. Stabilizační kroky určené na contacts/joints negativně ovlivňovaly orbital dynamics.
- **STATS sekce:** Time `D:HH:MM:SS`, Pixels, Objects, Connections, ∑P, ∑L, FPS.
- **FACTS sekce:** Fastest, Spinniest, Most momentum, Most ang. mom., Largest, Most massive — klik na `#ID` centruje kameru.
- **Home camera tlačítko** — návrat na (0, 0, zoom=32).
- **Číselné defaulty:** `G=1.0`, `H=1.0` (placeholder), `GRAVITY_EPSILON=1.5`, `GRAVITY_SUBSTEPS=4`, `SPAWN_LINVEL_MAX=0.1`.

### Kód

- `src/sim/gravity.ts` — kompletně přepsáno na `stepGravity()`: cache state do Float64Arrays, výpočet sil O(N²), kick-drift symplektický Euler, write back do RigidBody.
- `src/sim/diagnostics.ts` — total ∑P, ∑L, KE, těžiště. PE vrací `stepGravity` přímo.
- `src/sim/facts.ts` — championi, O(N) per tick.
- `src/sim/params.ts` — programové konstanty.
- `src/sim/physics.ts` — `spawnPixel(x, y)` zjednodušeno (params interní), přidáno `clear()` pro reset.
- `src/ui/App.svelte` — layout přepracován: STATS+FACTS na levo, SETTINGS+COMMANDS na pravo. ID jako klikací linky (kamera follow).
- `index.html` — title + locale `cs`.
- `vite.config.ts` — base path přepínaný env `GITHUB_PAGES`.
- `.github/workflows/deploy.yml` — auto deploy na Pages na push do main.

### Test/ověření

- ∑P, ∑L po 77 s simulace s 12 pixely: drží konstantní (oba pod 0.3 v absolutní hodnotě).
- Pixely tvoří kuželosečné dráhy kolem těžiště centrálního shluku — emergentní Keplerova dynamika.
- FPS = 60 stabilní s 12 pixely a 4 substepy.

### Kudos / Censure

**Kudos! uživateli:** Diagnostika s ∑P (klesá) a ∑L (diverguje 10⁵×) přesně lokalizovala viníka. Bez explicitních konzervačních readouts by byl bug tichý a my bychom slepě hledali jinde. Specifikace UI panelů jako konkrétní text byla extrémně užitečná — žádné rozhodování o detailech.

**Censure! sobě:** První pokus o fix (Rapier substepping) byl povrchní — předpokládal jsem truncation error, ne strukturální problém integrátoru. Mělo mi dříve dojít, že substepping pomáhá pouze symplektickému solveru, a Rapier ne nutně je. Druhá věc: zbytečně velké defaulty `G=2`, `linvel=2.0` v prvním pokusu o fázi 2 — měl jsem začít konzervativněji a postupně zvedat.

### Příště

- **Otevřená otázka pro fázi 3:** jak elegantně přepnout z manuálního integrátoru zpět na Rapier step(), aby přechod nebyl bolestný? Možná hybrid: gravitace ručně, joints + kolize Rapierem v jednom volání step po manuálním update?
- **Performance test:** kolik pixelů zvládne 60 FPS při O(N²) gravitaci + 4 substepy? Profilovat na 100, 500, 1000.
- **Vizualizace centroidu** jako křížek — uživatel to chce pro fázi 3.
- **Začít fázi 3** — slepování po straně. Designové otázky: detekce sousednosti (dotyk po straně), jeden vs. mnoho jointů na pixel, vznik composite objektu (datový vznik), update centroidu při změně topologie.
- **Pages deploy** ověřit (z konce sezení 1 stále nezaverifikováno).

## Sezení 3 — Plánování fáze 3, experimentální postup

### Diskuse

- Singularita gravitace: jakmile přijde kolizní systém + slepování přes constraint, minimální vzdálenost mezi pixely je geometricky vynucena na `r_min = U` (strana pixelu). `ε` (Plummer softening) zůstane v kódu jen jako numerická pojistka, ne jako fyzikální parametr — síla `F ∝ 1/U²` v close-encounter je konečná, ale velká, takže pojistka má smysl.
- Tři vrstvy otevřené pro fázi 3 (orchestrace Rapieru):
  1. Co Rapier `step()` interně dělá pro tělo bez external force / no joints / no contacts (gravity? damping? CCD?). Dosud spekulace, žádná čísla.
  2. Kam v cyklu zařadit `step()` — pre/post manual gravity, nebo PBD nadstavba s Rapierem jen na broadphase.
  3. Velikost driftu z kontakt-disipace vs. current float-roundoff drift z manuálního Eulera. Až bude přítomen, může být disipace dominantní a debata o symplektičnosti irelevantní.
- Postoj: postupné experimenty, každý generuje data, každý malý.
- Nápad uživatele: presety v COMMANDS panelu — tlačítko vymaže scénu, nastaví parametry, odstartuje. Volitelný auto-stop čas. Tlačítko **Export JSON** pro vložení stavu do chatu k analýze.

### Rozhodnutí

- **Singularita:** ε zůstane v kódu, ale schované jako numerická pojistka. Komentář v `params.ts` upravit (zatím TODO).
- **Pořadí experimentů:** E1 → E2 → E3 → E5.
  - **E1** Rapier baseline: 2 pixely, `gravity=(0,0)`, no joints, no contacts, manual linvel=0, 1000 steps. Měřím změny pos/linvel/angvel.
  - **E2** Rapier velocity damping: jako E1 ale s nenulovou linvel. Měřím tlumení (linearDamping default).
  - **E3** Rapier FixedJoint reakce: 2 pixely + joint, jeden kopnu, druhý reaguje. Měřím ∑P drift, joint stiffness.
  - **E5** Hybrid: manual gravity + Rapier `step()` bez jointů. Měřím ∑P/∑L drift vs. čistý manuální (sezení 2).
  - **E4** (contact response) a **E6** (close-encounter stability) → IDEAS, na pozdější sezení.
- **Architektura presetů:** experiment = preset. Žije v `src/sim/presets.ts`, UI v COMMANDS panelu, žádné samostatné `src/experiments/` ani Vitest. Export JSON = univerzální nástroj pro share stavu do chatu.

### Kód

**Preset/experiment infrastruktura:**
- `src/sim/presets.ts` — typy `Preset`, `PresetAPI`, `IntegrationMode`, `Modelshot`. Pole `PRESETS` s E1, E2, E3, E5. Funkce `buildModelshot()`.
- `src/sim/physics.ts` — `spawnPixelExact(x, y, vx, vy, r, rs, m)` pro deterministický spawn. Stávající `spawnPixel()` (random pro LMB) zachován jako tenký wrapper.
- `src/ui/App.svelte`:
  - Stav `paused`, `integration`, `currentPreset`, `stopAtTime`, `toast`.
  - Mode-aware loop: `manual` (stávající Euler), `rapier` (jen `world.step`), `hybrid` (oboje). Výchozí mód `manual` — LMB chování fáze 2 zachováno.
  - `Space` keydown = toggle pause; pauznutá simulace zmrazí čas, ale kamera + UI dál fungují.
  - Auto-stop: dosažení `preset.stopAtTime` → pauza + auto-export modelshotu do clipboardu.
  - Tlačítka **⏸ Pause**, **📋 Export JSON**, sekce **PRESETS** s aktivním zvýrazněním.
  - Toast notifikace (info "modelshot zkopírován" apod.).
  - `Reset scény` rozšířen o reset preset stavu.

**Co E3 NEDĚLÁ (vytknuto v komentáři presetu):** FixedJoint API ještě neexistuje (fáze 3 work). E3 je zatím pouze geometrický baseline (2 pixely u sebe, jeden v pohybu); reálné měření joint reakce vyžaduje doimplementovat joint API.

### Test/ověření

**E1 — Rapier baseline: PASSED (bit-identical).**
2 pixely v (-5, 0) a (+5, 0), vše nuly. Run 10 s, `rapier` mód, gravity=(0,0). Výsledek pos / linvel / angvel / r **bit-identical** k inputu. Rapier `step()` na klidu nepřidává žádný jitter (CCD, sleep, broadphase OK).

**E2 — Rapier velocity damping: PASSED, default damping = 0.**
2 pixely, jeden s `linvel=(1, 0)`, druhý s `angvel=0.5`. Po 10 s: `linvel` i `angvel` **bit-identical** se začátkem. Rapier `linearDamping` i `angularDamping` defaultně 0 — nic nemusíme vypínat.

**Boční nálezy z E2 (před opravou loopu):**
- `simTime` se akumuloval z `realDt` (wall-clock), ale Rapier integroval pevný `FIXED_DT = 1/60`. Drift ~0.3% mezi sim a Rapier časem. **Opraveno** Glenn Fiedler accumulatorem (`simTime += FIXED_DT` per fixed step). Po opravě `simTime = 600 × 1/60 ± float roundoff`.
- `r` (rotace) se v Rapieru **wrappuje do `(-π, π]`**. Pro reconstrukci „kolik otoček" by byl potřeba kumulativní counter, který zatím nemáme.
- **f32 drift v Rapieru:** `pixel.x` po 10 s s `vx=1` skončil na `4.99993` místo přesně `5.0`. Cumulative ~1e-7/krok × 600 = 6e-5. Není bug — `@dimforge/rapier2d-compat` používá f32 interně. **Pro E5 to znamená, že drift ∑P/∑L ≤ 1e-4 je „f32 floor", ne strukturální problém.**

**E5 vs. E5m — naive hybrid je BROKEN.**

Setup: 12 pixelů v D4 symetrii, G=1.0, ε=1.5, 60 s. E5 hybrid (manual + Rapier step), E5m manual only.

Konzervace (oba běhy):
- ∑P, ∑L: E5m drift ~1e-15/-1e-16 (**f64 epsilon**), E5 drift ~1e-12 (**f32 epsilon**). Hybrid je o 10⁴ worse jen kvůli tomu, že stav prochází Rapier WASM bridge ve f32.
- KE: 2.65 (manual) vs 2.68 (hybrid) — téměř shodné v okamžiku. Ne diskriminační (D4 symetrie).

Diskriminační nález: **inner pixely (start radius 1.5√2)** v manual módu **nepřešly přes počátek** (#8: (-1.5, -1.5) → (-1.36, -1.36), r = 1.92). V hybrid módu **přešly do opačného kvadrantu** (#8: (-1.5, -1.5) → (+2.06, +2.06)).

Hybrid prošel **víc oscilací** než manual za stejný `simTime`. **Confirmed**: naivní `manuální stepGravity + Rapier step()` integruje pos dvakrát (manuální drift + Rapier drift), simulace běží efektivně násobně rychleji.

### Rozhodnutí (konec sezení)

- **Architektura fáze 3 — odloženo** dokud nebudou jointy/kontakty implementované. Naive hybrid je nepoužitelný, ale konkrétní volba mezi (α) velocity-Verlet split, (β) save-zero-restore vel, (γ) Rapier step jen když jsou jointy/kontakty — vyžaduje reálný scénář s vazbou.
- Manual mode zůstává **default** pro všechno bez kontaktů. ∑P/∑L drift na f64 floor.
- Tři architektonické varianty (α/β/γ) zaznamenány do IDEAS.

### Performance test — naivní O(N²) gravita, manual mód, G=1

| N | Páry | FPS | Vyhodnocení |
|---|---|---|---|
| 100 | 4 950 | 60 | strop neviditelný |
| 500 | 124 750 | 60 | strop neviditelný |
| 1000 | 499 500 | 45 | viditelný frame drop |
| 2000 | 1 999 000 | 12 | nepoužitelné |

**Strop O(N²) padá kolem N≈700.** P1000→P2000 přechod je 4× víc párů, 3.75× pomalejší → **čistá O(N²) škála**, ~10 ns/op (rozumné pro V8 hot loop).

**Cíl ze sezení 1 „1000 pixelů @ 60 FPS"** je s naivní gravitací **nesplnitelný**. Spatial grid (uniform + cutoff) by měl dát ~10× speedup → 1000 @ 60 FPS s rezervou.

### Rozhodnutí (konec sezení)

- **Spatial grid promoted na prioritu [!]** — viz TODO.md. Bez něj je fáze 3 work limitovaná na N ≤ 500.
- Hybrid orchestrace pro fázi 3 (α/β/γ) zaznamenána v IDEAS, rozhodnutí odloženo na reálný joint scénář.
