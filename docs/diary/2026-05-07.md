# 2026-05-07

## Sezení 1 — Definice projektu, scaffold

### Diskuse

- Definice projektu: webová sandbox simulace dynamiky pixelů.
- Cíl: experiment a zábava, žádný fixní deadline ani feature scope.
- Konvence sezení: otázky a odpovědi se číslují `Q1, Q2…` / `A1, A2…`.
- Mód spolupráce: maximální skepticismus a kriticismus ze strany AI.
- Zatím bez `GLOSSARY.md` — pojmy se postupně budují, glosář vznikne, až bude třeba.

### Rozhodnutí

**Stack:**
- Vite + TypeScript (strict, `noUncheckedIndexedAccess`)
- Svelte 5 (UI panely, runes)
- Rapier 2D (`@dimforge/rapier2d-compat` 0.19.3) — rigid body fyzika
- WebGL2 + instanced quads (raw, bez render libs)
- gl-matrix (jediná math závislost)

**Klíčový pivot:** uživatel správně poukázal, že potřebujeme FW na 2D kolize. Původní návrh (custom XPBD + shape matching) byl over-engineered. Místo toho:
- Každý pixel = 1 rigid body s nativní rotací.
- Slepování = `FixedJoint` (později breakable).
- Pružnost = `DistanceJoint` se spring koeficientem.
- Rozbití = remove joint při překročení impulse threshold.

**Konvence:**
- Identifikátory anglicky, komentáře česky.
- Bez `T` prefixu (Delphi konvence se neaplikuje).
- `kebab-case.ts` filenames.

**Pixel struktura ve FVP:**
```ts
type Pixel = {
  id: number;
  body: RigidBody;   // Rapier drží x, y, vx, vy, r, rs
  m: number;         // konstantní 1, ale strukturně přítomné
};
```
Záměr: `m` připravený teď, abychom při fázi 4 nepřepisovali API.

### Kód

Scaffold projektu (Vite + Svelte 5 + TS), instalace deps, vytvořena adresářová struktura `src/{sim,render,input,ui}`. FVP implementace:
- `src/sim/physics.ts` — Rapier World wrapper, `spawnPixel(x, y)` s náhodnými počátečními rychlostmi a rotací.
- `src/render/gl.ts` — WebGL2 instanced renderer.
- `src/render/camera.ts` — 2D ortografická projekce, `screenToWorld` helper.
- `src/input/keyboard.ts` — set stisknutých kláves.
- `src/ui/App.svelte` — layout 3 sloupců, FPS + pixel counter, roadmap v pravém panelu.
- GitHub Actions deploy na Pages, MIT licence.

### Kudos / Censure

**Kudos! uživatel:** Včasné upozornění na potřebu kolizního FW — ušetřilo to týdny psaní vlastního shape matchingu. Skvělé řízení AI mimo over-engineering.

**Censure! sobě:** První návrh (XPBD + shape matching pro slepené clustery) byl ambiciózní v matematice tam, kde existující engine řeší problém přímočaře. Default to libraries first, custom solvers last.

## Sezení 2 — Fáze 2: gravitace + UI + pivot z Rapier integrátoru

### Diskuse

- Pixelová gravitace: párová Newton, centrální, nebo k těžišti? Uživatel zvolil **párovou**.
- Object dataset architektura — pre-computed centroid + cached state pro composite objekty (fáze 3+).
- Zkratka pro úhlovou rychlost objektu: `rs` (izomorfismus s `Pixel.rs`), ne VR/omega/W.
- Bug report: "stále se rozbíhají, gravitační prak". Diagnóza přes ∑P (klesá), ∑L (diverguje na ~10⁵) → integrátor je viník.
- Levý panel: STATS + FACTS (championi). Pravý panel: SETTINGS (G, H) + COMMANDS.
- `ε` (Plummer softening) je numerický regularizátor, ne fyzikální parametr → schováno z UI.
- Spawn linvel/angvel intervaly přesunuty do `params.ts` jako konstanty programu.

### Rozhodnutí

- **Solver pivot:** manuální symplektický Euler pro FVP, **Rapier integrátor obejit**. Rapier RigidBody zůstává jen jako kontejner stavu. Až ve fázi 3 (joints, kolize) se Rapier step() vrací — kontakty pak fyzicky brání slingshotům.
- **Důvod:** Rapier addForce + step v čistě gravitační simulaci porušoval Newton 3 a zachování úhlové hybnosti. Stabilizační kroky určené na contacts/joints negativně ovlivňovaly orbital dynamics.
- **STATS sekce:** Time `D:HH:MM:SS`, Pixels, Objects, Connections, ∑P, ∑L, FPS.
- **FACTS sekce:** Fastest, Spinniest, Most momentum, Most ang. mom., Largest, Most massive — klik na `#ID` centruje kameru.
- **Home camera tlačítko** — návrat na (0, 0, zoom=32).
- **Číselné defaulty:** `G=1.0`, `H=1.0` (placeholder), `GRAVITY_EPSILON=1.5`, `GRAVITY_SUBSTEPS=4`, `SPAWN_LINVEL_MAX=0.1`.

### Kód

- `src/sim/gravity.ts` — kompletně přepsáno na `stepGravity()`: cache state do Float64Arrays, výpočet sil O(N²), kick-drift symplektický Euler, write back do RigidBody.
- `src/sim/diagnostics.ts` — total ∑P, ∑L, KE, těžiště. PE vrací `stepGravity` přímo.
- `src/sim/facts.ts` — championi, O(N) per tick.
- `src/sim/params.ts` — programové konstanty.
- `src/sim/physics.ts` — `spawnPixel(x, y)` zjednodušeno (params interní), přidáno `clear()` pro reset.
- `src/ui/App.svelte` — layout přepracován: STATS+FACTS na levo, SETTINGS+COMMANDS na pravo. ID jako klikací linky (kamera follow).
- `index.html` — title + locale `cs`.
- `vite.config.ts` — base path přepínaný env `GITHUB_PAGES`.
- `.github/workflows/deploy.yml` — auto deploy na Pages na push do main.

### Test/ověření

- ∑P, ∑L po 77 s simulace s 12 pixely: drží konstantní (oba pod 0.3 v absolutní hodnotě).
- Pixely tvoří kuželosečné dráhy kolem těžiště centrálního shluku — emergentní Keplerova dynamika.
- FPS = 60 stabilní s 12 pixely a 4 substepy.

### Kudos / Censure

**Kudos! uživateli:** Diagnostika s ∑P (klesá) a ∑L (diverguje 10⁵×) přesně lokalizovala viníka. Bez explicitních konzervačních readouts by byl bug tichý a my bychom slepě hledali jinde. Specifikace UI panelů jako konkrétní text byla extrémně užitečná — žádné rozhodování o detailech.

**Censure! sobě:** První pokus o fix (Rapier substepping) byl povrchní — předpokládal jsem truncation error, ne strukturální problém integrátoru. Mělo mi dříve dojít, že substepping pomáhá pouze symplektickému solveru, a Rapier ne nutně je. Druhá věc: zbytečně velké defaulty `G=2`, `linvel=2.0` v prvním pokusu o fázi 2 — měl jsem začít konzervativněji a postupně zvedat.

- **Performance test:** kolik pixelů zvládne 60 FPS při O(N²) gravitaci + 4 substepy? Profilovat na 100, 500, 1000.
- **Vizualizace centroidu** jako křížek — uživatel to chce pro fázi 3.
- **Začít fázi 3** — slepování po straně. Designové otázky: detekce sousednosti (dotyk po straně), jeden vs. mnoho jointů na pixel, vznik composite objektu (datový vznik), update centroidu při změně topologie.
